# 말씀암송 인증방 PWA — 프로젝트 개요

## 서비스 개요

**말씀암송 인증방 PWA**는 소규모 교회·모임 구성원이 함께 말씀을 암송하고, 매일 녹음 또는 계수기로 인증하며, 서로의 인증 카드에 **아멘**으로 격려하는 모바일 웹 앱(PWA)이다.

> **핵심 메시지:** 함께 말씀을 암송하고, 매일 인증하며, 서로 아멘으로 격려하는 암송방

### 기술 스택 (명세 기준)

| 영역 | 기술 |
|------|------|
| Backend | Spring Boot |
| Database | Oracle SQL |
| Frontend | React PWA |
| 음성 파일 | 외부 스토리지 저장, DB에는 `AUDIO_URL`만 저장 |
| 삭제/취소 | 물리 삭제 대신 `STATUS = 'N'` 방식 |

---

## 핵심 사용자

| 사용자 | 설명 |
|--------|------|
| **일반 참여자 (MEMBER)** | 카카오 로그인 후 암송방에 참여하고, 매일 인증을 남기며 다른 사람의 인증 카드에 아멘을 남긴다. |
| **방장 (LEADER)** | 암송방을 생성하거나, 방 안에서 `ROOM_ROLE = 'LEADER'`로 관리한다. 암송 본문/공지를 등록할 수 있다. |
| **서비스 관리자 (ADMIN)** | `TB_MEMBER.ROLE = 'ADMIN'` — 서비스 전체 권한. MVP 범위에서는 별도 관리자 화면 미구현. |

---

## MVP 목표

1. 카카오톡 로그인으로 회원 가입·로그인
2. 암송방 생성 및 초대코드 입장
3. 암송 본문/공지 조회 (방장 등록)
4. 매일 녹음·계수기 인증 전송 및 취소
5. 날짜별·사람별 인증 피드 조회
6. 인증 카드에 아멘 토글
7. 오늘 내 인증 완료/미완료 상태 표시

### MVP 범위 밖 (추후 구현)

- 통계 화면 (개인 인증일 수, 연속 인증일, 방 참여율 등 — 원본 테이블에서 계산)
- 서비스 관리자 전용 화면
- 실제 카카오 OAuth·백엔드 API 연동 (MVP 프론트는 mock data 사용)

## 핵심 기능

| 기능 | 설명 | 주요 테이블 |
|------|------|-------------|
| 카카오 로그인 | `KAKAO_ID` 기준 회원 조회·등록 | `TB_MEMBER` |
| 암송방 생성 | 방 생성 + 생성자를 방장으로 등록 | `TB_ROOM`, `TB_ROOM_MEMBER` |
| 초대코드 입장 | `INVITE_CODE`로 방 입장 | `TB_ROOM`, `TB_ROOM_MEMBER` |
| 암송 본문/공지 | 방 안 암송 대상 말씀 등록·조회 | `TB_RECITATION_SECTION` |
| 하루 인증 카드 | 회원·방·날짜당 1개 카드 | `TB_RECITATION_CHECK_IN` |
| 녹음/계수기 인증 | 카드 안에 detail 다건 등록 | `TB_RECITATION_CHECK_IN_DETAIL` |
| 인증 취소 | detail `STATUS = 'N'`, 부모 STATUS 연동 갱신 | `TB_RECITATION_CHECK_IN_DETAIL`, `TB_RECITATION_CHECK_IN` |
| 아멘 | 하루 인증 카드 단위 토글 | `TB_RECITATION_AMEN` |
| 인증 피드 | 날짜 → 사람 → detail → 아멘 구조 | 전체 인증 관련 테이블 |

## 전체 사용자 흐름

```
[랜딩] → [카카오 로그인] → [홈: 내 암송방 목록]
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
            [암송방 생성]    [초대코드 입장]    [암송방 상세]
                                                    │
                    ┌───────────────────────────────┤
                    ▼                               ▼
            [암송 본문/공지]              [오늘 인증 상태]
                                                    │
                    ┌───────────────┬───────────────┤
                    ▼               ▼               ▼
            [녹음 인증]      [계수기 인증]    [날짜별 인증 피드]
                                                    │
                                                    ▼
                                              [아멘 토글]
```

### 흐름 상세

1. **비로그인 사용자** — 랜딩페이지에서 서비스 소개 확인 후 카카오 로그인
2. **로그인 후** — 홈에서 참여 중인 암송방 목록 확인, 새 방 생성 또는 초대코드 입장
3. **암송방 상세** — 현재 암송 말씀 확인, 오늘 인증 상태 확인, 인증 피드 열람
4. **인증** — 녹음 또는 계수기 방식으로 detail 추가 → 부모 CHECK_IN STATUS 갱신
5. **격려** — 다른 참여자의 하루 인증 카드에 아멘 (본인 카드 불가)
6. **MVP 이후** — 통계 화면에서 개인·방 단위 참여 현황 확인

## 프론트엔드 작업 시 지켜야 할 기준

1. **DB 명세 준수** — [`DB_SPEC.md`](./DB_SPEC.md)의 7개 테이블 구조를 임의로 변경하지 않는다.
2. **문서 우선** — 새 화면·컴포넌트 작업 전 [`SCREEN_FLOW.md`](./SCREEN_FLOW.md), [`UI_RULES.md`](./UI_RULES.md)를 먼저 확인한다.
3. **Mock data** — 백엔드 연결 전까지 [`MOCK_DATA_GUIDE.md`](./MOCK_DATA_GUIDE.md) 기준 mock data를 사용한다.
4. **인증 완료 판단** — `TB_RECITATION_CHECK_IN` row 존재 ≠ 완료. `STATUS = 'Y'`(유효 detail 1개 이상)일 때만 완료.
5. **아멘 기준** — `CHECK_IN_ID` 단위. detail 단위 아멘 없음. 본인 카드 아멘 불가.
6. **피드 구조** — 날짜 구분선 → 사람별 카드 → detail 목록 → 아멘.
7. **MVP 범위** — 통계·관리자 기능 등 MVP 이후 항목은 구현하지 않는다.
8. **애매한 사항** — DB 구조를 바꾸지 않고 문서에「추후 확인 필요」로 표시한다.
